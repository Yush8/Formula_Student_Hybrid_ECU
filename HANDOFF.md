# HCU V2 Firmware — Architecture & Handoff

Primer for the Formula Student Hybrid Control Unit firmware. Paste into a new chat
before asking for a feature, so it follows the established conventions.

## 1. Target
- **MCU:** STM32H745ZIT3, dual-core (CM7 + CM4), 2 MB dual-bank flash.
- **Clock:** VOS2 ceiling (~300 MHz on M7), **not** 480. HSE = 25 MHz **BYPASS** (MEMS osc); LSE = 32.768 kHz crystal.
- **All firmware is on CM7.** CM4 is the untouched generated stub — leave its HSEM boot handshake alone. Every pin/peripheral must be assigned to the **CM7 context** in CubeMX or no init is generated.
- **Toolchain:** STM32CubeIDE 2.1.1 + standalone CubeMX. Two sub-projects share one `.ioc`; all work is in `HCU_V2_CM7`.

## 2. Core philosophy
**CubeMX owns the project, peripherals, and `main.c`. Simulink owns ALL logic. Hand-written C is a thin bridge** that reads raw inputs into the model, steps it, and pushes raw outputs to hardware — it never decodes, scales, or makes vehicle decisions. "C is the postman; Simulink is the brain." Hand code survives regeneration only inside `/* USER CODE */` markers.

## 3. Must-not-break `.ioc` settings (verify after every CubeMX generate)
- **Power supply = "SMPS 1.8 V supplies LDO"** — anything else bricks boot.
- HSE 25 MHz BYPASS; LSE 32.768 kHz crystal; voltage scaling **VOS2**.
- USB 48 MHz from **HSI48 + CRS** (RC48 in clock tree, CRS sync = USB_FS) — independent of the PLL tree; don't retune PLLs for USB.
- FDCAN kernel clock = **HSE (25 MHz)**; FDCAN **AutoRetransmission = DISABLED** (control data must be fresh).

## 4. File / module structure (CM7)
**Generated by CubeMX, edit only in USER CODE markers:**
- `Core/Src/main.c` — init order in `USER CODE BEGIN 2`: `Params_Init(); Console_Init(); Can_Init(); Model_Init();`. Superloop (`USER CODE BEGIN 3`): `Console_Poll(); Can_Service();` + the ~100 Hz model stepper.
- `USB_DEVICE/App/usbd_cdc_if.c` — RX ring buffer + `CDC_Read()` in USER CODE; `CDC_Transmit_FS()` is the TX primitive.

**Hand-written modules (`Core/Src` + `Core/Inc`, outside regen):**
- `model_bridge.c/.h` — the adapter. Only file that includes both `main.h` and `HCU_V2_Simulink.h`. Exposes `Model_Init()`, `Model_Step()`.
- `params.c/.h` — parameter store: `g_params`, defaults, descriptor table, clamping, flash save/load. Single source of truth for tunables.
- `console.c/.h` — USB-CDC command console; thin UI over `params`.
- `can.c/.h` + `can1_messages.def` / `can2_messages.def` — dual-FDCAN demux/tx (see §13).

**Generated by Simulink (never hand-edit):** `Model/HCU_V2_Simulink_ert_rtw/HCU_V2_Simulink.{c,h}` etc.

**Off-target:** `hcu_console.py` — tkinter GUI front-end, same text protocol.

## 5. The model interface
Embedded Coder exposes three "doors":
- **Inputs `HCU_V2_Simulink_U`** — each root Inport = one named field; the bridge fills it.
- **Outputs `HCU_V2_Simulink_Y`** — each root Outport = one named field; the bridge dispatches it.
- **Parameters** — handled as extra inputs, not Simulink "parameters". Tunables live in firmware (`g_params`); the bridge copies them into `U` each step, so one struct serves model, console, and flash.

`HCU_V2_Simulink_initialize()` once at boot; `HCU_V2_Simulink_step()` each tick.

## 6. The bridge pattern
`Model_Step()` = **gather → step → dispatch**, at a fixed rate:
1. read raw hardware (pins, ADC, CAN snapshot) + params into `U`
2. `HCU_V2_Simulink_step()`
3. push `Y` to actuators (pins, CAN tx, …)

Hardware reality (which pin, active-low polarity) lives in the bridge; logic stays in Simulink. Conventions in use: button's raw electrical level passed to the model (TRUE = pin HIGH); active-low LED inverted in C (`Y.led ? RESET : SET`).

> The stepper is a **deterministic TIM6 time base** (`scheduler.c/.h`): the loop calls `Sched_StepDue()` each pass and runs one `Model_Step()` per elapsed tick. Exact 100 Hz, jitter-free, with an overrun counter. See §15.

## 7. Recipe — add a signal (GPIO/ADC sensor in / actuator out)
1. Draw an Inport/Outport in Simulink, name it, set its type; regenerate. A matching `U`/`Y` field appears.
2. Add **one line** to the bridge: `U.x = <read hw>;` (gather) or `<write hw> = Y.x;` (dispatch).

Same name in Simulink and C. Cost of a new signal = 1 port + 1 bridge line.

## 8. Recipe — add a tunable parameter
1. Simulink: add an Inport (`single` floats, `int32` ints, boolean→`int32` 0/1).
2. `params.h`: add a field to `Params_t`.
3. `params.c`: add a default in `PARAMS_DEFAULT` **and** a row in `TABLE[]` (name, type tag `P_F32`/`P_I32`, `&g_params.x`, lo, hi).
4. `params.c`: **bump `CONFIG_VERSION`** (struct shape changed — §10).
5. `model_bridge.c`: `U.x = g_params.x;`.

**Datatype rule:** the table reads each value via `void*` + type tag, so field size and tag must agree (`P_F32`=4-byte float, `P_I32`=4-byte int; bool stored as `int32_t` 0..1). Mismatch = reading neighbouring memory.

## 9. Config / flash persistence (the "blob")
One packed `ConfigBlob_t` = `{ magic, version, params, crc }` in a dedicated sector.
- **magic** distinguishes our data from blank/junk flash; **version** is the *config layout* version (catches valid-data-wrong-shape); **crc** (CRC32 over everything before it) catches corruption/half-writes.
- **Sector: bank 2, last sector (`0x081E0000`).** CM7 runs from bank 1; H7 read-while-write only works *across* banks, so writing config in bank 2 doesn't stall the running CPU. No CM7 linker change.
- `Params_Init()`: read → check magic/version/CRC → load if all pass, else compiled-in **defaults**; always **clamp** afterward. "Any doubt → safe defaults."
- `Params_Save()`: build blob, CRC, **erase then program**. Single copy → power-loss mid-save reverts to defaults next boot (safe). Treat `save` as a **pit/stationary action**.
- Usage: change value → `save`. Change struct shape → bump `CONFIG_VERSION`, reflash, `save`. `defaults` then `save` = factory reset.
- **Do not** call `SCB_InvalidateDCache_by_Addr` after the write — faulted on this board's MPU and isn't needed (config is read only at boot, cold cache).

## 10. USB CDC console
USB_OTG_FS, Device, Full Speed, behind an ADuM4160 isolator. DP/DM = PA12/PA11. Hardware VBUS sensing disabled; PA9 is plain GPIO reading isolated VBUS. Line-based text protocol over virtual COM: `help`, `list`, `get <name>`, `set <name> <val>`, `save`, `defaults`, `ping`. Firmware echoes chars (PuTTY local echo OFF). newlib-nano printf has **no `%f`** — format floats by hand (`f32_to_str`). TX = `CDC_Transmit_FS` with busy-retry. Only one program can hold the COM port at a time.

## 11. Simulink → IDE codegen workflow
- Embedded Coder, system target `ert.tlc`, Hardware board **None**, **Nonreusable function**, "Generate example main" **OFF**, "Generate code only" **ON**, fixed-step **discrete** solver at base rate, Hardware = ARM Cortex-M7.
- Point MATLAB's codegen folder into the project so it writes straight to `CM7/Model/`: `Simulink.fileGenControl('set','CodeGenFolder','<…>/CM7/Model','createDir',true)`.
- In CubeIDE the generated folder must be a **Source Location** (Project Properties → C/C++ General → Paths and Symbols → *Source Location*), **not just an Include path**. Include path resolves headers; Source Location is what compiles the `.c`. Missing it = `undefined reference` at link.
- Commit generated code; never hand-edit. Filenames are stable, so regen overwrites in place.

## 12. Key intricacies / lessons learned
- **Source Location vs Include Path** are different settings (§11). Note: `.def` files (X-macro lists) are *include-only* — they need the include path (Core/Inc) but **not** a Source Location.
- **Unbalanced brace** in `main.c` shows as `invalid storage class for function …` on the *next* function + `expected declaration … at end of input` — it's a missing `}`, usually at the `USER CODE BEGIN 3` / `while(1)` boundary.
- **Datatype size must match the table tag** (§8); **bump `CONFIG_VERSION`** whenever `Params_t` changes (§9); **no `%f`** under newlib-nano.
- After a build fix use **Run/Debug** (downloads); plain Build only compiles.
- **FDCAN bus-off recovery needs `HAL_FDCAN_Stop()` then `HAL_FDCAN_Start()`** — a bare `Start` fails because the HAL software state stays `BUSY` after a hardware bus-off (§13).
- **CubeMX regenerate corrupts `CM7/.project` linked resources (recurring — known CubeMX dual-core bug).** Two faces of one bug, both fail at the **link** step:
  - *HAL / USB library vanish* → wall of `undefined reference to HAL_*` / `USBD_*`. The regen flattened the shared-driver **folder-links** into dozens of loose **file-links**, which CDT's makefile generator ignores.
  - *FatFs duplicated* → `multiple definition of f_open / disk_read / …`. The regen added phantom file-links (`diskio.c ff.c ff_gen_drv.c syscall.c`) **next to** `Middlewares/Third_Party/FatFs/src`, so each compiles twice.
  - **Tell:** the failing build says **"Incremental Build … ~600 ms"** (link only) and `Debug/sources.mk` `SUBDIRS` is missing `Drivers/STM32H7xx_HAL_Driver/Src` and the USB-lib dirs — CDT won't regenerate the stale `Debug/` makefiles on an incremental build.
  - **Recovery (fast path):** restore the snapshot — `copy CM7\.project.known-good CM7\.project` — then **delete the whole `CM7/Debug` folder** (forces makefile regen) and Build. The snapshot has all driver/middleware sources as proper `type=2` folder-links pointing at `…/Src`, no FatFs strays.
  - **Recovery (by hand):** in `CM7/.project`, make HAL + USB-lib `type=2` folder-links to their `…/Src` dirs, and delete the 4 stray FatFs file-links; delete `CM7/Debug`; Build.
  - **Re-snapshot** `.project.known-good` whenever you intentionally add a peripheral that brings in a new HAL module (the linked-source set changes then).

## 13. CAN subsystem (DONE — use as the reference pattern)
2× FDCAN, **classic CAN, 500 kbit/s**, isolated transceivers. FDCAN1 = dash/ECU bus; FDCAN2 = tractive/inverter bus. No vehicle logic in C.

- **No hardware acceptance filters — deliberate.** Accept-all standard IDs into RxFIFO0; **software (bus, ID) linear-scan demux** in the ISR. Justified by few IDs + light bus load. The Rx ISR drains the *entire* 16-deep FIFO each time, so frame loss is effectively impossible (only its own brief `Can_Snapshot` IRQ mask could delay it). Loop rate does **not** affect frame loss — RX is fully decoupled in the ISR; loop rate only sets data freshness.
- **Adding a received message (the whole recipe):**
  1. one line in `can1_messages.def` / `can2_messages.def`: `CAN_MSG(MY_NAME, 0x123)` (no comma, no backslash);
  2. one line in `model_bridge.c` gather: `CAN_FEED(bus1, MY_NAME, my_port);`
  3. two Simulink inports: `my_port` (uint8, width 8) and `my_port_age` (uint32).
  The enum, routing table, and buffers are generated from the `.def` lists via include-based X-macros, so they **cannot desync** (this replaced a hand-kept enum+table that drifted).
- **Gather:** `Can_Snapshot()` masks only the two CAN Rx IRQs, copies each slot + `age_ms` (`UINT32_MAX` = never seen) + per-bus `bus_ok`. The model fails safe on stale `age_ms` / `bus_ok == false`.
- **Bus-off auto-restart:** `Can_Service()` runs each superloop pass — rate-limited recovery (retry 100 ms × 50 ≈ 5 s), then gives up leaving `bus_ok = false`. Guards against physical faults (loose connector), independent of bus load.
- **Diagnostics:** `g_can_stats` global (rx-lost + recovery counts) for **debugger watching only** — deliberately *not* fed into the model. Expected to read 0 at this load.
- **Tx:** `Can_Send(bus, id, data, len)` — returns false if the tx queue is full (caller decides policy). One line in the bridge dispatch; no scaling/endianness in C.

## 14. Robustness & usability roadmap (remaining)
1. **Deterministic scheduler — DONE (see §15).** TIM6 time base replaces the `HAL_GetTick` limiter.
2. **Independent AIR fail-safe (safety-critical).** Model-independent last resort in C: if the step loop stalls (watchdog) or the SDC monitor reads open → drive AIRs open regardless of model output. Precharge/AIR thresholds stay `const` in firmware, never in tunable config. Build to FS EV rules.
3. **Watchdog.** IWDG and/or external STWD100 (currently disabled). Kick **only** when the control loop is stepping healthily.
4. **A/B flash for config.** Two sectors, write+verify before marking current, so power loss mid-save keeps the previous good config.
5. **SD config mirror.** Human-readable `key value` export/import on microSD (reuse the console `set` parser); git-track setups per session.
6. **Console diagnostics (`stat`).** Loop rate, overrun count, `g_can_stats` (per-bus rx-lost/recoveries/last-seen), free stack — trackside without a debugger.
7. **Composite USB CDC + MSC (later).** Pull SD logs without removing the card.
8. **Dual-core split (later).** CM7 = control loop, CM4 = comms/logging, IPC via HSEM + shared SRAM.

**Peripherals still to bring up through the bridge:** SDMMC2 + FatFS (logging; an SD self-test currently sits in `main.c` USER CODE 2 — to be removed), SPI3 (ASM330LHHX IMU + INT), ADC (10k NTC thermistor), Timers/EXTI (2× opto-isolated wheel-speed), the SDC/AIR/precharge safety subsystem (highest rigour).

## 15. Scheduler / model time base (DONE)
`scheduler.c/.h` — the deterministic metronome for the control loop. Replaced the old `HAL_GetTick` rate limiter.

- **Ownership split:** **CubeMX** owns TIM6 (clock, prescaler/period, NVIC, IRQ routing) — the intended way, same as every other peripheral. **`scheduler.c`** owns only the policy (count ticks, run-latest). **Simulink** owns the math.
- **Pattern:** TIM6 overflows at the model base rate; its update interrupt does nothing but increment a tick counter (in `HAL_TIM_PeriodElapsedCallback`, guarded on `Instance == TIM6`). The superloop calls `Sched_StepDue()` each pass and runs **one** `Model_Step()` per elapsed tick. The cadence is hardware-exact (no drift with loop load), unlike the old `now - last >= 10` check.
- **CubeMX config (Timers → TIM6, on the CM7 context):** Prescaler **74**, Counter Period **9999**, Up mode; enable **TIM6 global interrupt** in NVIC1 at preemption priority **5** (below the FDCAN interrupts at 0, so CAN Rx always pre-empts the tick). Math: TIM6 clock = 75 MHz (APB1 37.5 MHz × 2, since `D2PPRE1 = DIV2`) → /75 = 1 µs tick → /10000 = 100 Hz = 10.000 ms. CubeMX then generates `MX_TIM6_Init`, the MspInit clock/NVIC, and `TIM6_DAC_IRQHandler` → `HAL_TIM_IRQHandler`.
- **Rate must match in three places:** Simulink fixed-step (`0.01 s`), the CubeMX TIM6 Counter Period, and `SCHED_RATE_HZ` in `scheduler.h` (currently `100`). Changing the base rate is rare; day to day you touch none of it.
- **Run-latest, no catch-up.** If the loop falls behind, `Sched_StepDue()` returns true once and counts the skipped ticks in `g_sched_overruns` — it never bursts steps to catch up (that would compress time, wrong for control). `g_sched_overruns == 0` means every deadline was met. Debug-only global, **not** fed to the model (same policy as `g_can_stats`); destined for the `stat` command (roadmap 6).
- **Wiring (`main.c`, all in USER CODE):** `Sched_Init()` runs last in `USER CODE BEGIN 2` (after `Model_Init()`, so the model is ready before the first tick) and just calls `HAL_TIM_Base_Start_IT(&htim6)`; superloop is `Console_Poll(); Can_Service(); if (Sched_StepDue()) Model_Step();`.
